Visión General del Proyecto

Advance Estate CRM es una plataforma de gestión de relaciones con clientes diseñada específicamente para RE/MAX Advance, franquicia inmobiliaria con sede en República Dominicana. El sistema centraliza la gestión de contactos, propiedades, pipeline de ventas, publicidad digital y comunicaciones de WhatsApp en una sola interfaz.

El proyecto combina un CRM web moderno con un agente de inteligencia artificial (Ava) que atiende automáticamente leads entrantes por WhatsApp, clasifica prospectos y sincroniza toda la información al CRM en tiempo real.

Objetivos del sistema

  • Centralizar la operación inmobiliaria: contactos, deals, propiedades, tareas y comunicaciones
  • Automatizar la atención inicial de leads CTWA (Click-to-WhatsApp Ads) mediante IA
  • Dar visibilidad de rendimiento a agentes y gerencia (KPIs, pipeline, ads)
  • Sincronizar automáticamente conversaciones de WhatsApp al CRM
  • Proporcionar una interfaz premium que refleje el posicionamiento de marca RE/MAX

Flujo de autenticación

El archivo proxy.ts intercepta todas las peticiones HTTP. Si el usuario no tiene sesión activa de Supabase Auth, redirige a /login. Las rutas /login y / son públicas.

// proxy.ts — Next.js 16
export async function proxy(request: NextRequest) {
  // Crea cliente Supabase SSR con cookies
  const { data: { user } } = await supabase.auth.getUser();

  const { pathname } = request.nextUrl;
  const isPublic = pathname === "/login" || pathname === "/";

  if (!user && !isPublic) {
    return NextResponse.redirect(new URL("/login", request.url));
  }
  return supabaseResponse;
}

Sincronización Ava → Supabase CRM

Esta fue una funcionalidad crítica del sprint: cada vez que Ava atiende un lead en WhatsApp, la conversación completa se sincroniza automáticamente a Supabase para que aparezca en el módulo de Conversaciones del CRM.

Diagrama de flujo

Lead envía msg WhatsApp
→
Meta webhook → Railway
→
Ava procesa con Claude
→
Responde al lead
↓
[fire-and-forget]
→
sync_contact_to_crm()
→
sync_message_to_crm() × 2

Función sync_contact_to_crm()

async def sync_contact_to_crm(phone, first_name="", last_name="",
    source="ctwa_ad", lead_classification="unqualified") -> str | None:
  """Upsert de contacto en Supabase. Retorna contact_id UUID."""
  async with httpx.AsyncClient() as client:
    resp = await client.post(
      f"{_SUPABASE_URL}/rest/v1/contacts?on_conflict=phone",
      headers={
        "apikey": _SUPABASE_KEY,
        "Authorization": f"Bearer {_SUPABASE_KEY}",
        "Content-Type": "application/json",
        "Prefer": "return=representation,resolution=merge-duplicates",
      },
      json={
        "phone": phone,
        "first_name": first_name,
        "last_name": last_name,
        "source": source,
        "lead_classification": lead_classification,
      }
    )
    data = resp.json()
    return data[0]["id"] if data else None

Función sync_message_to_crm()

async def sync_message_to_crm(contact_id, content,
    direction="inbound", is_automated=False) -> bool:
  """Inserta un mensaje en la tabla messages."""
  async with httpx.AsyncClient() as client:
    resp = await client.post(
      f"{_SUPABASE_URL}/rest/v1/messages",
      headers={ ... },
      json={
        "contact_id": contact_id,
        "content": content,
        "direction": direction,  # "inbound" | "outbound"
        "is_automated": is_automated,
      }
    )
    return resp.status_code == 201

Integración en main.py (fire-and-forget)

# En procesar_mensaje() — después de obtener respuesta de Claude
try:
  contact_id = await sync_contact_to_crm(phone=msg.telefono)
  if contact_id:
    await sync_message_to_crm(contact_id, msg.texto,
      direction="inbound", is_automated=False)
    await sync_message_to_crm(contact_id, respuesta,
      direction="outbound", is_automated=True)
except Exception as crm_err:
  logger.warning(f"CRM sync falló (no crítico): {crm_err}")
  # El WhatsApp flow NO se interrumpe
Diseño: Fire-and-Forget

El sync está envuelto en su propio try/except. Si Supabase no responde, Ava igual manda la respuesta al lead por WhatsApp. La confiabilidad del agente no depende de la disponibilidad del CRM.

Detalles técnicos del upsert

  • on_conflict=phone: Si ya existe un contacto con ese número, actualiza en lugar de crear duplicado
  • Prefer: resolution=merge-duplicates: Header de PostgREST para merge semántico
  • Prefer: return=representation: Retorna el registro upsertado (necesario para obtener el UUID generado)
  • Service Role Key: Se usa SUPABASE_SERVICE_ROLE_KEY (no anon key) para bypass de RLS en escrituras del agente

Problemas Resueltos & Fixes Técnicos

Fix #1 — Error 405 en todas las rutas del CRM

Síntoma

Todas las rutas del CRM retornaban HTTP 405 Method Not Allowed. El dev server mostraba el error al iniciar.

Causa raíz: Next.js 16 deprecó el archivo middleware.ts y la función exportada middleware(). La nueva convención es proxy.ts con función proxy().

Error en consola:

⚠ The "middleware" file convention is deprecated. Please use "proxy" instead.
Error: Both middleware file "./middleware.ts" and proxy file "./proxy.ts" are detected.
Please use "./proxy.ts" only.

Solución aplicada:

  • Creado proxy.ts con función proxy() (idéntica lógica al middleware anterior)
  • Eliminado middleware.ts vía PowerShell (el comando rm de bash era silencioso pero no confiable en Windows)
  • Limpiado caché .next: powershell Remove-Item .next -Recurse -Force
  • Reiniciado dev server: confirmado 307 → 200 en todas las rutas
# Comandos de limpieza usados
powershell -Command "Remove-Item -Path '.next' -Recurse -Force"
powershell -Command "Remove-Item -Path 'middleware.ts' -Force"
npm run dev

Fix #2 — getUser() duplicado en proxy.ts

El primer borrador del archivo proxy.ts tenía dos llamadas consecutivas a supabase.auth.getUser(). La primera fue removida con el Edit tool, dejando solo una llamada al final correctamente.

Fix #3 — Railway CLI token inválido

Los UUIDs proporcionados como "tokens de Railway" (56bb313c-..., 68880b62-...) eran en realidad Project IDs, no Personal Access Tokens. Railway CLI solo acepta tokens de tipo Personal Token en el env var RAILWAY_TOKEN. Las variables de entorno fueron añadidas manualmente vía el Dashboard de Railway (Raw Editor).

Fix #4 — Browser cacheando 405

Después de corregir proxy.ts y limpiar .next, curl confirmaba respuestas correctas (307→login, 200 en páginas públicas), pero el navegador seguía mostrando 405. Causa: el navegador tenía cacheada la respuesta de error. Solución: Ctrl+Shift+R (hard refresh) o abrir en ventana de incógnito.

Fix #5 — CLASSIFICATION_COLORS incompatible con Tailwind v4

La página de contactos usaba un objeto con strings de clases Tailwind (bg-red-100 text-red-700, etc.) generados dinámicamente. Tailwind v4 con JIT no puede detectar estas clases si no están presentes literalmente en el código. Se eliminó el objeto y se reemplazó con clases CSS utilitarias custom (.badge-hot, .badge-warm, etc.) definidas en globals.css.

Fix #6 — Template literals con clases dinámicas

Varios componentes usaban template literals para combinar clases con delays de animación: `card-glow p-5 ${delay}`. Esto causaba que Tailwind no purgara correctamente. Se cambió a concatenación simple: "card-glow p-5 " + delay.

Fix #7 — Variables Railway (entorno de producción)

Las variables SUPABASE_URL y SUPABASE_SERVICE_ROLE_KEY debían añadirse tanto en .env local como en Railway Dashboard para producción. El proceso fue:

  • Local: Editado whatsapp-agentkit/.env
  • Railway: Dashboard → Variables → Raw Editor → paste de ambas variables
  • Railway: Redeploy automático detectado por Railway al cambiar variables
